# AI 可靠性治理方法论（Reliability Governance）— 商用门槛

## 定位

当前体系有零散的防虚报（Dev git diff 自证、Reviewer 磁盘核对），但这是**点状防御**。商用需要的不是"某个角色自觉"，是**体系级兜底**——不管哪个角色犯错，体系都能拦住、能校验、能回滚。

本 skill 定义四道防线，让体系从"信任每个角色"升级为"不信任、必校验、可回滚"。

## 四道防线

### 防线 1：不可逆操作人工锁（Red Line Lock）

**原则：AI 不得未经人工确认执行任何不可逆操作。**

不可逆操作清单（人工锁，禁止 AI 自治）：

| 操作 | 为什么不可逆 | 谁来执行 | 人工锁 |
|---|---|---|---|
| 生产环境部署 | 影响真实用户 | Ops | ✅ 编排者确认 + 用户确认 |
| 数据库 schema 变更（DDL） | 删列/改类型丢数据 | DBA 设计，Ops 执行 | ✅ 用户确认 |
| 数据删除/清空 | 数据不可恢复 | Ops | ✅ 用户确认 |
| 生产配置变更（Nacos/配置中心） | 影响线上行为 | Ops | ✅ 用户确认 |
| 主分支 git push | 改写/污染历史 | Ops | ✅ 已有规则，保留 |
| 删除服务/下线模块 | 影响系统可用性 | Ops | ✅ 用户确认 |
| 模型部署到生产（MLOps） | 影响 AI 产品行为 | MLE 设计，Ops 部署 | ✅ 用户确认 |

**规则**：
- Ops/DBA/MLE 产出"操作计划 + 回滚方案"，但**不自动执行**
- 编排者复核计划合理性 + 回滚方案可行
- 用户显式确认"可以执行"后，Ops 才执行
- 执行后立即验证 + 留执行记录（谁、何时、改了什么、基于什么确认）

> 这条防线重定义了编排者"自治上报"的边界：**自治仅限可逆操作（代码改动、文档、测试），不可逆操作一律人工锁。**

### 防线 2：输出兜底校验（Defense-in-Depth）

**原则：任何角色的关键产出，必须有独立校验，不采信单方报告。**

当前已有 Dev 自证 + Reviewer/QA 磁盘核对，这是代码层的。兜底校验扩展到所有关键产出：

| 产出类型 | 产出者 | 独立校验者 | 校验内容 |
|---|---|---|---|
| 代码 | Dev | Reviewer + Security | 磁盘 diff 核对（已有） |
| 测试结论 | QA | 编排者抽查 | 关键 AC 是否真覆盖 |
| 数据模型/迁移 | DBA | Reviewer（规范）+ 编排者（幂等抽查） | 迁移脚本可重跑 |
| 性能结论 | Performance | 编排者 | 压测数据是否真实 |
| 部署/迁移方案 | Ops | 编排者 | 回滚方案是否可行 |
| 战略/架构决策 | EA/Strategy | 人工（用户） | 重大决策须人确认 |
| 度量数据 | Analyst/VSM | 编排者 | 数据来源是否真实 |

**核心原则（从不信任到校验）**：
- **产出者不自证**：Dev 不能只靠自己报告通过，Reviewer 必须独立核对。
- **校验者不抄作业**：Reviewer/QA 不抄 Dev 报告，必须自己跑 diff/grep/read。
- **编排者兜底抽查**：编排者在部署前做最终抽查（已有 fast 档核 1 处、standard 档核 2-3 处）。
- **重大决策上升到人**：架构/战略/预算类，编排者也不自治，上报用户。

### 防线 3：全链路可回滚（Rollback Everywhere）

**原则：每个 AI 动作都要可回滚。出了问题能退回安全状态。**

| 维度 | 回滚手段 | 谁负责 |
|---|---|---|
| 代码改动 | git checkout / git reset | Dev/Ops |
| 配置变更 | 配置版本化 + 回滚到上一版 | Ops |
| 部署 | 蓝绿/灰度 + 上一版本回滚 | Ops（须提前备回滚方案） |
| 数据库迁移 | 迁移脚本配套回滚脚本 | DBA（每个迁移必配回滚） |
| 模型部署 | 模型注册表 + 版本回滚 | MLE/Ops |

**强制规则**：
- 阶段5 部署前，Ops 必须先产出**回滚方案**（怎么退、退到哪、退了影响什么）。无回滚方案不得部署。
- 数据库迁移脚本必须配套回滚说明（哪怕是"手动恢复"也要写清步骤）。
- 部署用蓝绿/灰度，不走直接替换——出问题秒级回退。

### 防线 4：置信度评估与失败兜底

**原则：AI 产出要标注置信度，低置信度的必须人工复核或重做。**

**置信度信号**（编排者评估每个角色产出时关注）：
- 角色自己标注了不确定项（"此处假设 X，需确认"）→ 中置信
- 角色产出与上下文矛盾、或自相矛盾 → 低置信，打回
- 角色跳过了某些步骤（如 Reviewer 没提任何缺陷，可能没认真看）→ 低置信
- 角色多轮修改仍不稳定（每轮结论不同）→ 低置信，上报用户
- 角色产出与磁盘事实一致、覆盖完整、逻辑自洽 → 高置信

**失败兜底**：
- 角色产出不合格：打回重做，最多 2 轮（已有）
- 2 轮仍不达标：上报用户，不自作主张放行
- 阶段卡死（某角色反复失败）：编排者上报，不强行推进
- **诚实原则优先**：宁可慢、宁可上报，绝不掩盖失败、绝不假通过

## 与现有机制的关系（不是替代，是升级）

| 现有机制 | 本 skill 升级点 |
|---|---|
| Dev git diff 自证 | 扩展为"所有关键产出独立校验"（防线2） |
| 编排者部署前抽查 | 明确为兜底校验的一环 |
| 失败打回最多 2 轮 | 加上"置信度评估"判断何时该上报而非硬试 |
| Ops 回滚段 | 升级为"全链路可回滚 + 无回滚方案不部署"（防线3） |
| 自治上报规则 | 用不可逆操作清单重定义边界（防线1） |

## 商用预期管理（诚实声明）

本 skill 定义的是**治理规则与协议**（方法论层）。它能显著降低 AI 出错的风险——通过人工锁拦不可逆操作、独立校验破自证、回滚方案保安全、置信度评估防假通过。

但 AI 的可靠性是**概率性的，不是确定性的**——再严的治理也无法 100% 杜绝 AI 犯错。本 skill 把"不可控的 AI 出错"变成"可控的、可回滚的、可审计的"风险，这是商用可接受的前提，但不等于零风险。最终生产环境的把关责任仍在人。
