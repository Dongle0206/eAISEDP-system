# 项目群治理方法论（L2 项目群层）

## 触发约定

- `/项目群协同 <诉求>` —— 显式触发
- 战略确定后由 L3 移交

## 运作模式（PgM 是 L2 的编排者）

```
L3 战略确定（EA 架构蓝图 + 战略分析师优先级）
   ↓
PgM 拆解为项目群（识别子项目、依赖、优先级）
   ↓
PgM 派发：每个子项目派一个 team-orchestrator
   ├── 编排者A → 子项目A（带 L1 全套 13 角色）
   ├── 编排者B → 子项目B
   └── 编排者C → 子项目C
   ↓
PgM 协调：依赖、里程碑、资源、冲突
   ↓
各编排者完成 → PgM 汇总 → 交付验收
```

---

## 三层协同机制（重要——这是三层不变成孤岛的关键）

三层（L3 战略 / L2 项目群 / L1 战术）必须通过四个机制拉通，否则各跑各的、信息断层。

### 机制 1：共享数据契约（单一事实源）

所有产物存在一棵共享文档树，每层各写自己职责的目录，所有层都能读全局：

```
docs/
├── strategy/                    ← L3 写，L2/L1 读
│   ├── architecture/{blueprint, principles.md, roadmap.md, decisions/}
│   ├── tech-radar.md
│   └── portfolio-priority.md    ← L2 PgM 排期输入
├── program/                     ← L2 写，L3 读(验证)/L1 读(执行约束)
│   ├── roadmap.md               项目群路线图
│   ├── dependencies.md          跨项目依赖图 ← L1 BA 必读
│   ├── milestones.md            里程碑跟踪 ← 核心（见机制3）
│   ├── resources.md
│   └── metrics/                 交付效能（VSM）
├── standards/                   ← L2 写，L1 读(执行)/L3 读(回验)
│   ├── engineering-manual.md    ← L1 Reviewer/QA 必读
│   ├── quality-gates.md         ← L1 门禁角色必读
│   └── claude-templates/        各子项目 CLAUDE.md ← L1 注入
├── 需求文档|过程执行文档|测试报告/case-*/  ← L1 写，L2 读(进度)
└── 过程跟踪文档/changelog.md    全局变更日志
```

**铁律**：这是单一事实源。每层只写自己目录，但跨层读取是强制的——编排者启动前必读 L3 架构原则 + L2 标准/依赖。

### 机制 2：双向上下文传导

```
L3 EA/战略                  L2 PgM/标准/VSM              L1 编排者+13角色
    │                            │                            │
    │ ── 下行：架构原则 ────────→ │ ── 下行：标准/依赖/排期 ─→ │ 执行约束
    │                            │                            │
    │ ←── 上行：架构冲突 ─────── │ ←── 上行：进度/风险/效能 ── │ 执行反馈
    │                            │                            │
    │ ←── 上行：原则失效反馈 ────────────────────────────────── │ 越级回灌(架构类)
```

- **下行**：通过派发协议显式打包传递（PgM 派发编排者时，把架构原则+工程标准+依赖约束+里程碑要求一起传下去）。
- **上行**：通过事件驱动上报（机制4）+ 产物落盘位置自动可见。
- **越级回灌**：编排者发现架构原则不可行时，标注"架构类冲突"可越级直报 EA（见机制 4 的冲突路径）。

### 机制 3：项目群里程碑追踪协议（核心——PgM 管住多子项目的关键）

**里程碑双层结构**：项目群里程碑（PgM 管）= 所有子项目里程碑（各编排者管）的聚合。PgM 不直接追踪 case，追踪子项目里程碑；编排者负责把子项目里程碑拆成 case 级阶段产出。

**milestones.md 标准格式**（PgM 维护，所有编排者读写）：

```markdown
# 项目群里程碑跟踪 — {项目群名}

## 里程碑 M1：{名称}（目标日期 YYYY-MM-DD）
| 子项目 | 负责编排者 | 当前状态 | 阻塞项 | 预计达成 |
| 子项目A | 编排者A | 🟡进行中(阶段3) | 无 | 按时 |
| 子项目B | 编排者B | 🔴阻塞 | 依赖A的接口契约 | 延期2天 |
| 子项目C | 编排者C | 🟢已达成 | - | - |

## 跨项目协同点（容易出事的地方）
- M1 前：子项目A的接口契约必须先于子项目B开发确定（强依赖）
- M1 前：子项目C的DB schema必须稳定（影响A和B）
- M2 前：所有子项目集成测试通过（PgM主持）
```

**状态约定**：🟢已达成 / 🟡进行中(当前阶段) / 🔴阻塞(阻塞原因) / ⚪未启动

### 机制 4：事件驱动 + 定期扫描双重上报

编排者向 PgM 的进度上报采用**双重机制**：

**A. 事件驱动（编排者主动报）**：
- 每完成一个阶段产出 → 编排者更新 milestones.md 自己那行状态（🟢/🟡/🔴 + 当前阶段 + 阻塞项）
- 遇阻塞/预计延期 → 立即更新 + 在风险日志记录
- 不依赖 PgM 主动问——这是编排者的强制义务

**B. 定期扫描（PgM 主动巡检）**：
- PgM 每个周期（按项目群节奏，如每日/每迭代）扫描 milestones.md 全表
- 识别 🔴 阻塞项和预计延期 → 主动介入协调
- 弥补事件驱动可能漏报的盲区

**双重保障**：正常推进靠事件驱动（实时），全局把控靠定期扫描（兜底）。

## 冲突上交路径（逐级为主 + 架构例外越级）

```
L1 编排者遇冲突
   ↓
能自己解决？ → 解决，记录
   ↓ 不能
判断冲突类型：
   ├─ 普通（资源/进度/接口契约）→ 上交 L2 PgM 裁决
   │    └─ PgM 能解决？→ 解决
   │         ↓ 不能且涉架构
   │         上交 L3 EA
   │
   └─ 架构类（架构原则冲突/技术选型违背原则）→ 【越级】直报 L3 EA
        └─ EA 裁决，必要时更新架构原则集 + 写新 ADR
```

**规则**：
- 默认逐级上交（编排者→PgM→EA），保持指挥链清晰。
- **架构类冲突可越级直报 EA**——因为架构决策时效性强，经 PgM 转发会延误且 PgM 不具备架构裁决权。
- 越级上报时必须**同时通知 PgM**（保持信息同步，PgM 需据此调整排期）。
- 资源/进度/接口类冲突**必须经 PgM**，不能越级（EA 不管这些）。

## 派发协议（PgM 派发子项目给编排者时必须传递）

PgM 派发时，打包传递上层约束，编排者据此执行：

1. 子项目目标与范围边界
2. **架构约束**（来自 EA 的架构原则，路径 `docs/strategy/architecture/principles.md`）
3. **工程标准**（来自标准负责人，路径 `docs/standards/`）
4. **依赖项**（依赖哪个子项目的什么产出，来自 `docs/program/dependencies.md`）
5. **里程碑要求**（本子项目在 milestones.md 里负责的里程碑）
6. **本项目 CLAUDE.md**（技术栈/规范，来自标准负责人）

编排者收到后，启动前**必读**上述所有约束文件，执行中遵循。

## PgM 核心动作

1. **战略拆解**：把 L3 战略拆成项目群（识别子项目、依赖、优先级、启动时序）
2. **派发**：每个子项目派一个 team-orchestrator，打包传递上层约束
3. **依赖管理**：维护依赖图，依赖链风险预警，接口契约先行协商
4. **里程碑协调**：维护 milestones.md，双重上报，🔴 介入协调
5. **资源调度**：13 角色是能力池，平衡各子项目投入，关键路径优先
6. **冲突裁决**：普通冲突 PgM 裁决，架构冲突转 EA
