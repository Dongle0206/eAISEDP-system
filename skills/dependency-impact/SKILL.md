# 全局依赖影响分析方法论（横向跨层）

## 定位

L1 BA 的变更影响评估是**单 case 内**的。大型项目里，一个底层模块改动会波及上游 N 个项目——需要**全局依赖影响分析**。本 skill 由 PgM（跨项目协调）和 BA（单项目评估升级）调用。

## 依赖图模型

维护项目群级**依赖图**（`docs/knowledge/dependency-graph.md`）：

```
节点 = 模块/服务/接口/数据实体
边 = 依赖关系（A 调用 B、A 引用 B 的数据）

依赖类型：
- 强依赖：接口契约变更必影响（编译/运行失败）
- 弱依赖：数据格式变更可能影响（需验证）
- 运行时依赖：部署顺序约束
```

## 影响传播分析

当某节点变更，做**影响传播分析**：

```
变更节点 X
   ↓
沿依赖图反向追溯：谁直接依赖 X？（一阶影响）
   ↓
继续追溯：谁依赖一阶影响节点？（二阶影响）
   ↓
直到无更多影响 或 达到项目群边界
   ↓
产出影响清单：受影响项目/模块/接口 + 影响类型（强/弱）+ 建议动作
```

## 产出：影响分析报告

```markdown
# 变更影响分析 — {变更描述}

## 变更源
- 节点：{模块/接口}
- 变更类型：契约变更/数据格式/行为变更

## 一阶影响（直接依赖）
| 项目 | 模块 | 影响类型 | 必须动作 |
| 子项目A | 用户服务 | 强依赖 | 改调用方式 |

## 二阶影响（间接依赖）
| 项目 | 模块 | 影响类型 | 建议动作 |

## 跨项目协调建议（给 PgM）
- 子项目A 需在 X 变更前适配
- 子项目B 可并行，但测试需覆盖集成点
```

## 使用场景

- **PgM**：底层服务改造前，评估波及面，排期协调
- **BA**：单项目变更涉及共享模块时，升级到全局评估
- **EA**：架构演进决策时，评估迁移影响面
