# 企业架构方法论（L3 战略层）

## 触发约定

- `/企业架构 <诉求>` —— 显式触发
- 战略规划流程中由战略分析师协同激活
- L1 编排者上报跨项目架构冲突时激活

## 运作模式

```
企业架构诉求输入
   ↓
EA 评估现状（读取战略文档、现有架构、技术债）
   ↓
产出四层架构蓝图（业务/应用/数据/技术）
   ↓
定义架构原则集（向下传导给 L2 标准负责人 → L1 编排者）
   ↓
制定演进路线（当前态 → 目标态，分阶段）
   ↓
重大决策写 ADR 入库
   ↓
（如有跨项目冲突）裁决 + 必要时更新原则集
```

## 与 L2/L1 的协同

- **向下传导**：EA 的架构原则，由 L2 方法论标准负责人写入工程标准（CLAUDE.md / 质量门禁），L1 编排者遵循。
- **接收上报**：L1 编排者遇到架构冲突或需偏离原则时，上报 PgM → EA 裁决。
- **与战略分析师协同**：EA 出架构方案，战略分析师算 ROI 和可行性，共同支撑立项决策。

## 人机边界

EA 自主产出架构方案；涉及全企业级技术栈更替、核心系统废弃、大额架构重构预算，保留人工确认点。

## 架构冲突裁决（接收 L1 越级上报）

L1 编排者遇架构类冲突（架构原则不可行/技术选型违背原则）时，可**越级直报 EA**（不经 PgM）。EA 接收后：

1. 评估冲突——是原则有问题，还是编排者理解有误？
2. 裁决——给出明确结论（遵循原则 / 原则需调整 / 特例豁免）
3. 若原则需调整 → 更新 `docs/strategy/architecture/principles.md` + 写新 ADR + 通知 L2 标准负责人同步工程标准 + 通知 PgM 调整排期
4. 记录冲突与裁决到 ADR 库（沉淀为组织知识）

**越级上报只限架构类**。资源/进度/接口类冲突必须经 PgM，EA 不接收这类上报。
