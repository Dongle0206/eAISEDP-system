# 合规审计方法论（横向跨层）

## 定位

L1 Security 查的是**代码安全**（越权/注入/漏洞）。本 skill 查的是**合规审计链**——企业级有等保/GDPR/PCI-DSS/行业合规要求，需要审计追溯能力。

## 合规维度

| 合规类型 | 关注点 | 常见要求 |
|---|---|---|
| 等保（网络安全等级保护） | 系统安全等级、日志留存、访问控制 | 日志留存6月+、操作可审计 |
| GDPR / 个人信息保护法 | 个人数据收集/处理/存储/删除 | 数据最小化、删除权、处理记录 |
| PCI-DSS（支付） | 支付卡数据安全 | 加密存储、访问审计、合规评估 |
| 行业合规 | 医疗/金融/政务等特殊要求 | 按行业规范 |

## 审计追溯链

为每个合规要求维护**审计追溯链**：

```
合规要求（如"用户数据删除可审计"）
   ↓ 映射到
控制措施（代码实现：删除接口 + 日志记录 + 操作人留痕）
   ↓ 映射到
证据（日志样本、代码位置、测试验证）
   ↓ 定期
合规自检 → 发现缺口 → 整改 → 闭环
```

## 合规检查清单（通用基线）

- [ ] 敏感数据是否加密存储/传输
- [ ] 用户数据删除请求是否有审计记录
- [ ] 操作日志是否满足留存时长要求
- [ ] 访问控制是否有审计（谁在何时访问了什么）
- [ ] 数据收集是否符合最小化原则（是否超范围采集）
- [ ] 第三方数据共享是否有合规依据

## 产出

- 合规自检报告（`docs/compliance/audit-report.md`）
- 审计追溯链文档（`docs/compliance/traceability.md`）
- 合规缺口与整改清单（`docs/compliance/gaps.md`）

## 使用场景

- **Security**：代码安全评审时，超出代码层面涉及合规的，调用本 skill
- **EA**：架构设计时，合规要求作为架构约束输入
- **方法论标准负责人**：把合规要求转成工程标准（如日志留存策略）
- **定期合规审计**：按合规周期激活，做全面自检
