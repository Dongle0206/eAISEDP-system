# 成本治理方法论（Cost Governance）— 商用可持续

## 定位

单 case 跑下来，18 个角色、每个派生、每个注入上下文+经验记忆+CLAUDE.md，token 消耗是线性的，规模化（每天几十上百 case）会失控。本 skill 让成本可计量、可控、可优化。

**诚实边界**：真实 token 计量依赖运行时框架上报。本 skill 定义计量口径、控制策略、优化手段（方法论层），框架照此实现。

## 三大成本控制手段

### 1. Token 预算与计费

**计量口径**（每个角色每次派生都要记录）：

| 维度 | 记录内容 |
|---|---|
| case-ID | 属于哪个 case |
| 角色 | 哪个角色消耗的 |
| 输入 token | 注入的 prompt + 上下文 + 经验记忆 |
| 输出 token | 角色产出 |
| 模型 | 用了哪个模型（关联 model-registry） |
| 成本 | 按 model-registry 的单价折算 |
| 时间 | 何时 |

**预算机制**：
- 每个 case 设 token 上限（默认按档位：fast 档 ≤ X、standard 档 ≤ Y，由 CLAUDE.md 配置）
- case 级累计消耗达 80% 预算 → 编排者预警
- 达 100% → 编排者停止派生新角色，评估是升预算还是裁范围
- 项目群级（PgM）有总预算，VSM 监控项目群成本分布

### 2. 模型动态分级降级（与 model-registry 联动）

不是所有任务都要用配置的默认档位。编排者可按任务复杂度**临时降级**：

| 降级场景 | 原档位 | 降为 | 条件 |
|---|---|---|---|
| 简单 CRUD 代码 | Dev structured | mechanical | 任务复杂度低、模板化 |
| 简单文档产出 | Ops mechanical | 已最低，不降 | — |
| 标准测试用例 | QA structured | mechanical | 照 AC 反推、无复杂边界 |
| **门禁角色** | Reviewer/Security reasoning | **❌ 不降** | 降了门禁失效 |
| **架构/战略角色** | EA/Strategy reasoning | **❌ 不降** | 降了决策质量崩 |

**铁律**：reasoning 档不降（降了质量崩），structured 按任务复杂度判断，mechanical 已最低。降级是临时策略，标注有效期，质量风险高的任务回调原档。

### 3. 上下文按需注入裁剪

最大的 token 浪费在"全量注入"。按需裁剪：

| 上下文类型 | 全量注入 vs 按需 |
|---|---|
| 项目 CLAUDE.md | 按角色相关章节注入（Dev 注入编码规范段，不注入部署段）|
| 经验记忆 | 已有分层策略（<10全量/≥10只high+tag命中）|
| 依赖文档 | 只注入与本 case 相关的依赖项，不注入全依赖图 |
| 架构原则 | 只注入与本子项目相关的原则 |
| 历史 case | 不注入，只给检索索引 |

**裁剪规则**：编排者派生角色时，按角色职责裁剪上下文到"本次任务必需"的最小集，不无脑全量塞。

## 成本优化优先级（从高到低）

1. **上下文裁剪**（收益最大——多数浪费在全量注入）
2. **模型分级降级**（mechanical 干的绝不用 reasoning）
3. **避免无效派生**（fast 档不派不必要角色——已有触发规则）
4. **经验记忆复用**（避免重复踩坑浪费轮次）

## 成本可见性

编排者在 case 交付时附**成本报告**：各角色 token 消耗、模型分布、是否超预算。VSM 汇总项目群成本趋势，异常飙升告警 PgM。
